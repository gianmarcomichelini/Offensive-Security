## JWT Bypass via Flawed Signature Verification

### Vulnerability Overview
Building upon the fundamental concepts of JSON Web Token manipulation, this specific challenge introduces a severe architectural flaw within the domain of cryptographic verification, given that the backend server, while indeed attempting to verify token signatures, explicitly trusts the client to dictate the actual verification method by dynamically reading the alg (algorithm) field located within the token's header. Consequently, by intentionally manipulating this header to specify the none algorithm, an attacker successfully instructs the backend parsing library to skip all cryptographic mathematics entirely, a process which effectively renders the token unsecured and thereby allows for an absolute and unconstrained payload manipulation.

### Exploitation Workflow
Regarding the exploitation workflow, the initial phase requires one to perform a Header Modification by intercepting a valid JWT assigned to the standard session and extracting the first segment, which constitutes the header, so that it can be decoded from Base64url in order to change the specified algorithm to none, thereby resulting in a JSON structure exactly like `{"alg": "none", "typ": "JWT"}`, which must subsequently be re-encoded back into a strict Base64url format. Following this, the process of Privilege Escalation mandates the extraction and decoding of the second segment, known as the payload, where one must modify the internal claims to escalate the privileges to match the administrative role, an action achieved by altering the identity to `"username": "admin"` or by setting an administrative flag such as `"is_admin": 1`, before re-encoding this newly modified payload into Base64url.

The subsequent crucial phase is the Signature Amputation, which involves assembling the final forged token by concatenating the modified header and the modified payload, and because the none algorithm explicitly dictates that no cryptographic signature exists, it is imperative to completely delete the third segment of the JWT, keeping in mind that one must strictly retain the final trailing period to satisfy the backend parser's structural logic, ultimately resulting in the specific format of `EncodedModifiedHeader.EncodedModifiedPayload.`. Finally, the Bypass Execution is finalized by submitting this precisely crafted token within the Cookie or Authorization header to a restricted endpoint, whereby the server will read the manipulated header, bypass the signature check entirely, and grant full administrative access based entirely on the forged payload claims, ultimately allowing one to retrieve the final flag.
