## KeyVault - JKU Injection


> x Exam (22)

The practical execution of a dynamic key retrieval attack utilizing a cloud based mocking service requires a methodical approach, ensuring strict cryptographic alignment between the forged token and the adversarial infrastructure. The following workflow details the exact steps necessary to replicate the successful authorization bypass utilizing Webhook.site as the external payload delivery mechanism.

> **Adversarial Telemetry** is the critical advantage provided by webhook platforms, allowing the offensive practitioner to observe incoming network requests in real time, thereby confirming whether the target application has successfully parsed the forged routing instructions and executed the out of band connection.

### Phase 1: Infrastructure Provisioning and Configuration

The engagement begins with the rapid provisioning of the external infrastructure. The practitioner navigates to the Webhook.site platform, which immediately provisions a unique and globally routable Uniform Resource Locator upon initial connection. This specific address will serve as the target destination for the backend application.

To ensure the target server properly parses the retrieved cryptographic materials, the practitioner must configure the platform to serve a valid JSON response. Within the platform interface, the practitioner accesses the response editing capabilities. It is essential to explicitly define the `Content-Type` header as `application/json` within the custom headers section. Subsequently, the practitioner inserts the fully structured JSON Web Key Set into the response body text area. ### Phase 2: Token Interception and Header Forgery

With the external payload staged, the practitioner transitions to the local interception proxy to manipulate the session token. The practitioner captures a legitimate HTTP request directed toward the restricted administrative interface and isolates the JSON Web Token residing within the Cookie header.

Utilizing the specialized JSON Web Token editor, the practitioner focuses entirely on subverting the cryptographic routing mechanisms located within the token header. First, the practitioner injects a new `jku` parameter, assigning it the absolute string value of the unique Uniform Resource Locator generated by the Webhook platform. Immediately following this injection, the practitioner must update the `kid` parameter within the token header to perfectly synchronize with the `8ae3be71-b804-483f-89c1-4260a44de5ab` identifier embedded within the staged JSON Web Key Set.

  "kid": "8ae3be71-b804-483f-89c1-4260a44de5ab",

### Phase 3: Privilege Escalation and Cryptographic Signing

The routing instructions are now complete, requiring the practitioner to define the final objective of the exploit. The practitioner shifts focus to the payload segment of the decoded token, locating the access control claims. The `role` parameter is manipulated, elevating the assigned value from the standard user context to `admin`.

To finalize the forgery, the practitioner utilizes the extension interface to mathematically sign the entire token structure using the local, adversarial private RSA key corresponding to the public key hosted on the webhook. It is paramount during this specific operation to instruct the signing utility to preserve the manually crafted header, preventing the tool from overwriting the injected routing instructions with default cryptographic parameters.

### Phase 4: Execution and Telemetry Verification

The practitioner transmits the newly forged, cryptographically signed HTTP request to the target application server. Simultaneously, the practitioner monitors the Webhook.site dashboard interface.

The successful execution of the vulnerability is visually confirmed when a new incoming GET request populates within the webhook interface, demonstrating that the backend Python server successfully parsed the `jku` header and initiated an outbound connection to retrieve the adversarial key set. Concurrently, within the Burp Suite interface, the target application responds with an HTTP 200 OK status code and returns the restricted administrative HTML content, proving that the retrieved key was successfully utilized to validate the forged signature and execute the requested privilege escalation.
