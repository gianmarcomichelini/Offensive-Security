## JWT authentication bypass via jwk header injection
The laboratory exercise focuses on a specific authentication bypass vulnerability arising from the improper validation of the JSON Web Key parameter within a session management token. The target environment utilizes a JWT based mechanism that actively supports the `jwk` header parameter to embed verification keys directly, yet critically fails to verify if these provided keys originate from an explicitly trusted source.

> **Embedded JWK Attack Vector** leverages a critical trust failure where the verifying application dynamically accepts and utilizes an adversary supplied public key embedded directly within the token header, enabling the successful validation of arbitrarily forged cryptographic signatures.

### Initial Access and Reconnaissance

The practical execution commences within the Burp Suite environment, requiring the preliminary installation of the JWT Editor extension from the BApp store to facilitate advanced cryptographic token manipulation. The practitioner initiates the engagement by authenticating against the target application utilizing the provided standard user credentials, specifically the `wiener` account. Upon successful authentication, the resulting HTTP GET request directed toward the `/my-account` endpoint is intercepted and forwarded to the Repeater module for isolated, repetitive analysis. To establish a baseline understanding of the access control mechanisms, the target path within the intercepted request is manually modified to target the `/admin` endpoint, which predictably results in an authorization failure, confirming that the administrative interface is strictly protected and requires privilege escalation.

### Cryptographic Key Generation

The exploitation methodology mandates the generation of rogue cryptographic materials to forge a mathematically valid signature later in the process. The practitioner navigates to the dedicated JWT Editor Keys interface located within the primary application tabs. A new RSA key pair is generated by selecting the appropriate automated function, which creates the required cryptographic parameters without necessitating manual key size specification, and the resulting key pair is securely saved within the local extension repository for subsequent signing operations.

### Payload Manipulation and Token Forgery

Returning to the previously modified administrative request residing within the Repeater module, the practitioner utilizes the specialized JSON Web Token interface provided by the extension to inspect the decoded token structures seamlessly. The critical modification occurs within the payload segment of the token, where the subject claim, designated by the `sub` parameter, is altered from the standard user identifier to the target `administrator` identifier.

### Exploitation and Objective Execution

The core vulnerability is exploited by utilizing the automated attack functionality embedded within the extension, specifically selecting the Embedded JWK vector. The previously generated adversarial RSA key is selected during this automated process, prompting the tool to dynamically construct a `jwk` object containing the public key components, inject it into the token header, update the corresponding `kid` parameter to ensure structural alignment, and cryptographically sign the forged payload utilizing the adversarial private key.

```
{
  "kid": "8df82987-a226-4b9e-bd71-344cb89dcfa3",
  "typ": "JWT",
  "alg": "RS256",
  "jwk": {
    "kty": "RSA",
    "e": "AQAB",
    "kid": "8df82987-a226-4b9e-bd71-344cb89dcfa3",
    "n": "vXw_qM2u..._wX9Q"
  }
}
```

The manual alternative to this automated process requires the practitioner to manually append the `jwk` JSON object into the header and manually synchronize the key identifier values before computing the signature. The transmission of this newly forged token to the target server results in a successful authorization bypass, granting unrestricted access to the administrative panel. The final objective is achieved by parsing the server response to identify the specific endpoint responsible for user management, discovering the deletion route, and transmitting a final forged HTTP request to the `/admin/delete?username=carlos` endpoint to successfully neutralize the specified target account.
